## 1. R1：formal120（旧早停规则）

**起点**：SFT checkpoint（`qwen3_1.7b_gsm8k_lora...eosfix2`合并后的actor起点，GRPO LoRA从0初始化）。

**配置要点**：

```text
train_batch_size=4, rollout_n=8, ppo_epochs=2, ppo_mini_batch_size=16
learning_rate=1e-6（早期），total_training_steps=120
早停规则（旧）: val_em连续40步未刷新全局最佳即停止
```

**结果**：

| step | -1 | 39 | 79（旧规则早停点） | 99 | 109（best） | 119 |
|---:|---:|---:|---:|---:|---:|---:|
| EM | 0.47 | 0.56 | 0.55 | 0.64 | **0.65** | 0.63 |

旧早停规则在 step 79 触发（EM 曾在 39 步达到 0.56 后回落又回升），但从 checkpoint-79 恢复后 EM 继续在 99/109 步刷新新高，证明 step79 是**误停**而非收敛。

**问题**：早停规则只看"是否刷新全局最佳"，不看"近期是否有恢复趋势"，导致提前放弃仍在改善的训练。

**下一轮调整**：

1. 早停规则整改为"全局最佳耐心 + 最近3个验证点趋势斜率"，只有耐心耗尽且无恢复趋势才真正停止（见 45 号文档 9.3 节）。
2. 用完整 `checkpoint-119` 延长总预算到500步，观察更长曲线是否继续改善。

## 2. R2：formal500（旧框架，延长自 checkpoint-119）

**起点**：`checkpoint-119`（R1终点），沿用同一 output/CSV，续跑 step 120–499。

**配置要点**：与 R1 相同结构，但沿用旧版早停/KL/格式监控（未整改）。

**结果**：

| step | 119 | **169（best）** | 199 | 209 | 218（停止点） |
|---:|---:|---:|---:|---:|---:|
| greedy EM | 0.63 | **0.67** | 0.55 | 0.61 | signal guard 触发停止 |
| actor-reference kl_loss | - | 0.0386 | 0.0744 | - | 0.2335 |
| rollout format | - | - | - | - | 0.594 |
| response长度 | - | - | - | - | mean 214 / max 256 |

**问题（旧框架的直接缺陷，见 45 号文档 12.2 节）**：

1. KL 保护指标接错：用局部 `approx_kl`（相对 old policy）而非累计 `reference_kl`（相对 SFT reference）判断"是否远离 reference"。
2. `kl_loss_coef=0.001` 长期约束过弱，无自适应增强机制。
3. 无格式正确答案仍能拿到约0.3正奖励，给模型留下放弃格式的捷径。
4. overlong 惩罚只看1200字符，从未触发；无token上限命中率、EOS率、截断惩罚。
5. greedy eval format 较高掩盖了随机rollout的尾部退化，无固定sample@n格式检查。

format 与 response length 相关系数约 -0.77，与 actor-reference KL 相关系数约 -0.745，指向"偏离reference → 输出变长/截断 → 格式尾部退化"这条链路。

**下一轮调整**：

```text
分离 reference KL / update KL → reference KL 滑窗预警与硬保护
→ 自适应 KL 系数 → 严格单一 #### 格式门控
→ token 长度/EOS/截断惩罚 → 固定随机 sample@n 验证
→ 异常 rollout 原文、source index 和 bucket 留档
```

不从 step218 强行续跑，改为用 `checkpoint-169`（best）建立新分支，显式允许训练目标变化。

## 3. R3：repair_probe_from169（整改后首次验证）

**起点**：`checkpoint-169`，新分支，显式允许目标变化（reward v2 + 新KL guard + sample@n验证）。**但无条件加载了checkpoint-169的旧optimizer state**（这一点后来在R4中被识别为混杂因素）。

**配置要点**：50步上限探针，reward已改为v2（正确性优先于格式）。

**结果**：实际完成42步，step211因reference KL连续3窗口超过0.1安全停止。

| 层次 | 结论 |
|---|---|
| 工程监控与保护 | 通过：指标语义、预警、自适应系数、异常留档、硬停止均生效 |
| 能力提升目标 | 未通过：EM未超过起点0.67，reference KL最终失控 |

update_kl最大仅0.000113（远低于阈值），说明不是单次PPO更新爆炸，而是许多局部小更新长期累积成reference偏离。自适应KL系数`0.005→0.0075→0.01125`确实在调整，但第一次增强在step199，第二次在step209，介入太晚。

**问题（已分级，见45号文档第二部分第7节）**：

- 已确认直接原因：reference KL连续超限，guard正确停止。
- 强证据支持：KL penalty长期占policy loss比例过小（0.16%~3.83%）；自适应KL10步间隔+1.5倍增益反应过慢。
- 尚未隔离的共同因素：`ppo_epochs=2`重复放大小prompt batch梯度方向；`lr=5e-6`可能偏高；每步仅4个独立prompt导致跨题梯度方差大；**继承了checkpoint-169的旧optimizer动量（污染因素）**。
- 已排除：数据损坏、坏数据顺序、格式reward压过数学reward、单次PPO爆炸、NaN/OOM。

这是一次混合修复实验（同时改了reward、KL配置、验证形态、guard，又继承旧optimizer），不是干净的单变量A/B，因此**不能声称找到唯一主因**，必须转入控制变量实验。

**下一轮调整**：设计7分支控制变量矩阵（O1旧optimizer隔离、P1 PPO epoch、L1学习率、K1/K2 KL系数与间隔、B1 prompt数），所有分支统一重置optimizer作为实验卫生条件。

## 4. R4：首轮控制变量 Gate10→Screen30（7分支）

**起点**：统一从`checkpoint-169`的同一份LoRA权重出发，`weights_only`加载并重置optimizer（O1例外，故意保留旧optimizer作为隔离对照）。

**分支矩阵**：

| 分支 | 唯一变量 |
|---|---|
| C0 | 无（fresh optimizer + lr5e-6，干净基线） |
| O1 | 继承checkpoint-169旧optimizer |
| P1 | ppo_epochs 2→1 |
| L1 | learning_rate 5e-6→3e-6 |
| K1 | kl_loss_coef 0.005→0.02 |
| K2 | adaptive_kl_interval 10→2 |
| B1 | 独立prompt 4→8（保持optimizer update次数不变） |

**结果（Screen30, 30步，见`screen30_decision_cn.md`）**：

| 分支 | KL斜率趋势(0-9/10-19/20-29) | rollout EM 首→末 | final greedy/sample EM | 决定 |
|---|---|---|---|---|
| C0 | 0.00021/0.00101/0.00033 | 0.6625→0.6000 | 0.63/0.35 | 保留为对照 |
| O1 | 0.00050/0.00097/0.00147 | 0.6562→0.5781 | 0.59/0.35 | 淘汰：KL持续加速且能力更差 |
| P1 | 0.00035/0.00045/0.00077 | 0.6469→0.6406 | 0.65/0.375 | 暂停：能力尚可但KL加速，sample截顶达0.10 |
| **L1** | 0.00045/0.00060/0.00058 | 0.6469→0.6219 | 0.64/0.40 | **进入Confirm50：KL未持续加速且能力优于C0** |
| K1 | 0.00074/0.00065/0.00071 | 0.6688→0.6219 | 0.60/0.3625 | 淘汰：提高系数未改善实际KL，能力下降 |
| K2 | 0.00022/0.00104/0.00200 | 0.6750→0.6156 | 0.68/0.4125 | 淘汰：系数过早降到0.001，漂移加速 |
| B1 | 0.00033/0.00082/0.00268 | 0.6406→0.5781 | 0.62/0.325 | 淘汰：KL加速最明显，计算量翻倍无收益 |

**问题/发现**：

1. O1支持"继承旧optimizer会放大KL漂移和能力下降"，但不足以证明是唯一主因。
2. P1（降PPO epoch）能降低单次update KL并保持rollout能力，但不能阻止reference KL尾部加速。
3. K1（提高初始KL系数）没有改善实际reference KL——说明问题不是"系数绝对值太小"这么简单。
4. K2（更快响应）反而因为系数过早降到下限而加速漂移——响应频率不是越快越好。
5. B1（增加独立prompt）没有解决问题且计算量翻倍。
6. 唯一满足Confirm50最低门槛的是L1（降学习率），但其KL绝对表现仍未优于C0，只能称为候选，不是已证实的解法。

**下一轮调整**：只将C0（对照）和L1（唯一候选）续到Confirm50（50步）。

## 5. R5：Confirm50（C0 vs L1，seed42）

**起点**：C0/L1各自的`checkpoint-29`，full resume续到step49。

**结果（见`confirm50_results_cn.md`）**：

| 指标 | C0 | L1 | L1-C0 |
|---|---:|---:|---:|
| 40–49步KL均值 | 0.049950 | 0.047127 | -0.002823 |
| 40–49步KL斜率 | +0.000942 | -0.000094 | -0.001036 |
| step49 KL | 0.054634 | 0.039070 | -0.015564 |
| raw warning越界次数 | 2 | 1 | -1 |
| final greedy/sample EM | 0.60/0.3375 | 0.64/0.3875 | +0.04/+0.05 |

L1在40–49步KL不再加速（斜率转负），能力、rollout信号、sample格式和截顶均优于C0；C0则在40步后出现新一轮尾部漂移（KL均值从0.038升到0.050），sample格式率跌到0.8625。

**问题（L1还未证明的事）**：

- 没有证明EM能持续上升，终点0.64仍低于零步基线0.67。
- 没有证明3e-6学习率在其他seed中仍优于5e-6（单seed结论）。
- 没有证明旧optimizer、PPO重复更新、学习率之间不存在交互。
- 没有证明现有自适应KL控制器具备"主动拉回"能力（L1的KL回落更可能来自较小学习率下的随机轨迹差异，而非控制器warning前主动增强）。

**下一轮调整**：C0/L1新增seed=123、2026，做三seed复验，判定标准见下轮。

## 6. R6：三seed复验（C0/L1，seed 42/123/2026）——判定"不晋级"

**起点**：新seed从同一checkpoint-169权重、`weights_only`加载、fresh optimizer开始（不能从seed42的checkpoint-49继续），各自走完整的Gate10→Screen30→Confirm50。

**过程插曲**：seed=2026在旧signal guard下曾在step11因窗口重叠误停（"连续3个滑动窗口"实际共享90%数据）；guard整改为非重叠窗口+mixed仅预警后，重新执行30步→50步续训（详见45号文档12.1.1–12.1.3节）。

**三seed预注册判定结果**：

| 门槛 | 支持seed数 | 是否通过 |
|---|---:|---|
| 尾10步KL均值更低 | 3/3 | 通过 |
| 尾10步KL斜率不更高 | 2/3 | 通过 |
| greedy retention不低于C0 | 2/3 | 通过 |
| sample retention不低于C0 | 3/3 | 通过 |
| rollout EM尾10步不低于C0 | 2/3 | 通过 |
| **sample格式率同时`>=0.90`且`>=C0`** | **1/3** | **不通过** |
| sample截顶率同时`<=0.10`且`<=C0` | 2/3 | 通过 |
| hard KL越界为0 | 3/3 | 通过 |

**总体结论：不晋级**。L1把尾部KL均值平均降低0.012467，是可复现的稳定因素，但没有形成同时满足KL、能力、rollout和格式安全的完整训练配方。按预注册规则，不直接启动L2或K3，也不从SFT重新训练，不能挑选单个最高EM的seed继续。

seed=123的L1格式率0.8875只比门槛少1条合法回答，seed=2026的0.8750少2条——差距很小，足以提出"扩大sample评估以区分抽样噪声与真实退化"的新诊断，但不能反向推翻"不晋级"的既有结论。

**下一轮调整**：把L1（3e-6）确立为新对照组（不再用5e-6的C0作对照），在此基础上分别测试L2剂量（继续降学习率）和K3控制器（禁止KL系数降到初始值以下）。

## 7. R7：g7lk——L2/K3剂量与控制器实验（当前最新一轮，2026-07-23~24）

**起点**：`checkpoint-169`同一份LoRA，seed=42，以三seed确认后的**L1（lr=3e-6）为新对照**，不再用5e-6的C0直接对照。

**分支**：

| 分支 | 相对L1的唯一变化 |
|---|---|
| l1 | 无（本阶段控制组） |
| l2 | 学习率 3e-6→2e-6 |
| k3 | `adaptive_kl_min_coef` 0.001→0.005（禁止系数降到初始值以下） |

**Screen30晋级判定**（见`g7lk/_orchestration/s30/decision.md`）：

| 门槛 | l2 | 判定 | k3 | 判定 |
|---|---:|---|---:|---|
| 尾10步KL均值差`<=+0.01` | -0.000335 | 通过 | -0.001820 | 通过 |
| 尾10步KL斜率差`<=+0.0005` | +0.000132 | 通过 | +0.000277 | 通过 |
| greedy retention差`>=-0.02` | +0.0100 | 通过 | **-0.0300** | **不通过** |
| rollout EM尾10步差`>=-0.03` | +0.0031 | 通过 | -0.0188 | 通过 |
| sample格式率`>=0.90` | 0.9250 | 通过 | 0.9375 | 通过 |
| sample截顶率`<=0.10` | 0.0875 | 通过 | 0.0750 | 通过 |

l2全部通过进入Confirm50；k3因greedy retention超标未晋级——更强KL约束确实压低了KL，但代价是能力下降超出预注册边界，保留`checkpoint-29`不再续训。不运行L2+K3组合，避免收益/代价来源无法区分。

**Confirm50结果（l2 vs l1，各50步，见`g7lk/_orchestration/c50/`）**：

| 指标 | l1（对照） | l2 | l2-l1 |
|---|---:|---:|---:|
| 尾10步reference KL均值 | 0.047127 | 0.038083 | -0.009044 |
| 尾10步KL斜率 | -9.4e-05 | -0.000311 | -0.000217 |
| step49 KL | 0.03907 | 0.031658 | -0.007412 |
| **final greedy EM** | 0.64 | 0.64 | **0.0000** |
| **final sample EM** | 0.3875 | 0.3875 | **0.0000** |
| rollout EM尾10步 | 0.5594 | 0.5875 | +0.0281 |
| sample格式/截顶 | 0.9125/0.075 | 0.9125/0.0875 | 持平/+0.0125 |

**核心发现**：

1. **学习率剂量方向已打平**：l2的KL轨迹进一步优于l1（尾部KL均值再降0.009），但greedy EM和sample EM相对l1**逐位小数完全相等**——不是接近，是数值相等。继续降低学习率的能力边际收益已归零，KL越来越稳但换不来更高正确率。
2. **长度偏置假说未复现**：l1/l2/k3三条轨迹的`val_response_len_mean`全程稳定在140–146 token，没有随训练推进上涨。之前担心的"formal500式的长度膨胀"只在旧框架（R2）出现过，reward v2整改（长度/截断惩罚+严格格式门控）在checkpoint-169之后的所有受控续训中持续压住了长度，这次新数据进一步验证了这一点。
3. **checkpoint-169起点的0.67可能是评估噪声**：100题固定验证集下，比例估计95%置信区间半宽约0.094（`1.96×√(0.65×0.35/100)`），而所有continuation分支的终点EM都稳定收敛在0.61–0.66区间，没有一条重新达到或超过0.67。这个差距落在单点评估的抽样噪声范围内，不能排除0.67本身只是一次幸运高点。

**下一轮调整（见第8节）**：不再在"学习率/KL系数"这条轴上继续加密网格；优先验证0.67是否为噪声，再考虑此前提出但未执行的独立prompt数（R4/B1曾用不同方式测试，未采用等更新预算设计）和无放回epoch sampler方向。

## 8. 当前待改进方向与下一轮建议

> R8 的具体执行方案（分阶段顺序、预注册门槛、资源估算）见 [`44_grpo_r8_next_round_experiment_plan_cn.md`](44_grpo_r8_next_round_experiment_plan_cn.md)。该方案基于代码调研发现：B1 原本已经控制了 optimizer update 次数（真正未控制的是总 rollout 量），且 `datasets/gsm8k_sft/test.parquet`（1319题、与训练集零重叠）可直接用于零成本验证 0.67 是否为评估噪声，因此 R8 第一步不是训练新分支，而是先做免训练的噪声核验。

### 8.1 已经打平/证伪的方向（不建议继续投入）

| 方向 | 证据 | 结论 |
|---|---|---|
| 继续降低学习率（5e-6→3e-6→2e-6） | l1→l2 EM完全不变，只改善KL | 边际收益已归零，停止在此轴加密 |
| 提高KL系数下限（K1/K3两轮验证） | K1：系数升高不改善实际KL；K3：能力下降超标 | 已两次证伪，不应再试更高系数 |
| 加快自适应KL响应频率（K2） | 系数过早降到下限，加速漂移 | 响应更快不等于约束更强，已证伪 |
| 继承旧optimizer（O1） | KL漂移放大，能力更差 | 已确认为负面因素，新分支应始终fresh optimizer |
| loss长度偏置修复 + 从SFT重开 | 长度在R7三条轨迹中稳定140-146token，未复现膨胀 | 当前证据不支持；应作为技术债务保留（见`42_grpo_deviation_from_standard_cn.md`），不是当前僵局的解法 |

### 8.2 尚未验证、值得优先尝试的方向

1. **验证0.67是否为评估噪声**（成本最低，建议最先做）：对现有100题eval集做bootstrap置信区间，或者換更大的held-out集合重新评估checkpoint-169、l1、l2的best/final checkpoint，判断0.61-0.67之间的差距是否有统计学意义。这一步不需要重新训练，直接复用已有checkpoint即可。
2. **R4/B1的干净版本**：R4中的B1（独立prompt 4→8）在"未控制总rollout数和optimizer update次数"的情况下被淘汰，这本身混杂了"prompt数"和"计算量"两个变量。48号文档9.2节已经指出这一点，应设计"用梯度累积等更新预算"的干净版本重新测试跨题梯度方差假说。
3. **无放回epoch sampler**：45号文档9.4节指出，120 step独立随机采样只覆盖5759条数据的约8%。换成无放回epoch sampler能大幅提升数据覆盖率，这个方向在整个v7系列中从未实际执行过。
4. **PPO epoch=1 与 L1 的交互**（P1在Screen30暂停，未证伪）：P1单独测试时能力保持较好但KL仍加速；48号文档指出"只有P1的多步证据仍有价值时，新增L1+ppo_epochs=1交互实验"，这个组合尚未做过。

### 8.3 不建议现在做的事

- **不建议**修复loss长度偏置后从SFT重新训练：没有证据支持这是当前僵局的成因，重新训练成本高且会浪费checkpoint-169已验证的0.47→0.67提升。
- **不建议**继续在学习率或KL系数轴上加密网格：两个方向都已经验证到边际收益归零/转负。
- **不建议**直接选取任意最高EM的checkpoint作为"最终模型"：0.67很可能是噪声，应先做置信区间验证再决定交付哪个checkpoint。
